# Caching Strategies for Next.js at Scale

Caching is not a single switch in Next.js. It is a layered system that spans the browser, CDN, the Next.js data cache, and your own infrastructure. This guide shows how to use each layer intentionally, with modern patterns that keep performance high and data correct.

## The Caching Layers You Actually Control

Think in layers. Each layer solves a different problem and has a different invalidation story.

- Browser and CDN cache: fastest delivery, best for static and semi-static assets.
- Next.js data cache: memoized fetch results with configurable revalidation.
- Full route cache: pre-rendered HTML for static or ISR pages.
- External cache (Redis or KV): cross-request cache for data not owned by Next.js.

## Data Cache: The Default Workhorse

In the App Router, `fetch` calls are cached by default for the lifetime of a build or until you specify revalidation. Use `cache: "no-store"` for user-specific or highly volatile data.

```ts
// lib/data/catalog.ts
export async function getCatalog() {
  const res = await fetch("https://api.example.com/catalog", {
    next: { revalidate: 300, tags: ["catalog"] },
  });

  if (!res.ok) {
    throw new Error("Failed to fetch catalog");
  }

  return res.json() as Promise<Array<{ id: string; name: string }>>;
}
```

```ts
// lib/data/user.ts
import { cookies } from "next/headers";

export async function getUserProfile() {
  const session = cookies().get("session")?.value;

  const res = await fetch("https://api.example.com/me", {
    cache: "no-store",
    headers: { Authorization: `Bearer ${session}` },
  });

  if (!res.ok) {
    throw new Error("Failed to fetch profile");
  }

  return res.json() as Promise<{ id: string; name: string }>;
}
```

## Full Route Cache and ISR

Static generation makes the whole route cacheable. Add `revalidate` to turn it into ISR and keep pages fresh without sacrificing speed.

```tsx
// app/blog/[slug]/page.tsx
import { getPostBySlug, getAllSlugs } from "@/lib/data/blog";

export async function generateStaticParams() {
  const slugs = await getAllSlugs();
  return slugs.map((slug) => ({ slug }));
}

export const revalidate = 120;

export default async function BlogPostPage({ params }: { params: { slug: string } }) {
  const post = await getPostBySlug(params.slug);
  return <article>{post.title}</article>;
}
```

## Segment Config: Make Freshness Explicit

Use segment config when you need global behavior for a route segment.

```ts
// app/metrics/page.tsx
export const dynamic = "force-dynamic";
export const fetchCache = "force-no-store";

export default async function MetricsPage() {
  const data = await fetch("https://api.example.com/metrics", { cache: "no-store" });
  const json = await data.json();
  return <pre>{JSON.stringify(json, null, 2)}</pre>;
}
```

## Request Memoization: Free Wins in a Single Render

Within a single server render, React memoizes identical `fetch` calls. This prevents accidental duplication in the same request.

```tsx
// app/account/page.tsx
import { getUserProfile } from "@/lib/data/user";

export default async function AccountPage() {
  const profile = await getUserProfile();
  const profileAgain = await getUserProfile();

  return (
    <div>
      <h1>{profile.name}</h1>
      <p>Same data, no extra request: {profileAgain.name}</p>
    </div>
  );
}
```

## Invalidation: The Part That Actually Matters

Caching is only safe if invalidation is precise. Use tags for data-level invalidation and paths for route-level invalidation.

```ts
// app/admin/actions.ts
"use server";

import { revalidatePath, revalidateTag } from "next/cache";
import { updateCatalog } from "@/lib/data/admin";

export async function updateCatalogAction(formData: FormData) {
  await updateCatalog(formData);

  revalidateTag("catalog");
  revalidatePath("/shop");
}
```

## On-Demand Revalidation (Webhooks)

If your CMS can send a webhook, revalidate only the content that changed.

```ts
// app/api/revalidate/route.ts
import { NextResponse } from "next/server";
import { revalidateTag } from "next/cache";

export async function POST(req: Request) {
  const { secret, tag } = await req.json();

  if (secret !== process.env.REVALIDATE_SECRET) {
    return NextResponse.json({ ok: false }, { status: 401 });
  }

  revalidateTag(tag);
  return NextResponse.json({ ok: true, revalidated: tag });
}
```

## External Cache: Redis for Cross-Request Reuse

When multiple routes need the same data across requests, an external cache gives you more control.

```ts
// lib/cache/redis.ts
import Redis from "ioredis";

const redis = new Redis(process.env.REDIS_URL);

export async function getCached<T>(key: string) {
  const value = await redis.get(key);
  return value ? (JSON.parse(value) as T) : null;
}

export async function setCached<T>(key: string, value: T, ttlSeconds: number) {
  await redis.setex(key, ttlSeconds, JSON.stringify(value));
}
```

```ts
// lib/data/pricing.ts
import { getCached, setCached } from "@/lib/cache/redis";

export async function getPricing() {
  const cacheKey = "pricing:v2";
  const cached = await getCached<Array<{ sku: string; price: number }>>(cacheKey);

  if (cached) return cached;

  const res = await fetch("https://api.example.com/pricing", { cache: "no-store" });
  if (!res.ok) throw new Error("Failed to fetch pricing");

  const data = (await res.json()) as Array<{ sku: string; price: number }>;
  await setCached(cacheKey, data, 300);
  return data;
}
```

## A Practical Caching Policy

- User-specific data: `cache: "no-store"`
- Shared read-heavy data: `next: { revalidate: 60-600 }`
- Content pages: ISR with `revalidate` and tags
- Admin updates: call `revalidateTag` and `revalidatePath`
- Cross-route shared data: Redis or KV cache

## Final Takeaway

Great caching in Next.js is deliberate. Use the data cache for most reads, ISR for pages, tags for precise invalidation, and external caches only when cross-request reuse is required. This is how you keep performance high without losing correctness.
